# WebAuthn Passwordless Authentication - Implementation Report

## Abstract

This report documents the implementation and testing of a WebAuthn-based passwordless authentication service. The system implements the full WebAuthn protocol for both user registration (attestation) and authentication (assertion), with comprehensive cryptographic verification and multi-device support. The service supports cross-browser and cross-device credential reuse through stable RP ID and origin configuration.

## 1. Introduction

### 1.1 WebAuthn Overview

WebAuthn (Web Authentication) is a web standard that enables passwordless authentication by leveraging public-key cryptography. Users register authenticators (hardware security keys, platform authenticators like Windows Hello, mobile biometrics) and log in using cryptographic signatures rather than passwords.

### 1.2 Project Scope

This implementation provides:

- Complete attestation ceremony (registration) with server verification
- Complete assertion ceremony (authentication) with server verification
- Multi-device and cross-browser credential support
- Secure session management with counter-based replay protection
- Sample web UI for register/login/dashboard flows

### 1.3 Key Goals

- ✓ Implement strict WebAuthn protocol compliance
- ✓ Support multiple authenticators per user
- ✓ Enable cross-device credential reuse
- ✓ Provide comprehensive cryptographic verification
- ✓ Maintain production-ready code quality

## 2. Implementation Details

### 2.1 RP Configuration

**RP ID (Relying Party Identifier):**

The RP ID is the domain suffix that uniquely identifies the relying party. For WebAuthn to function correctly:

- For localhost/development: `RP_ID = "localhost"`
- For production: `RP_ID = actual.domain.com` (must be a registrable domain)
- Browser validates that `RP_ID` is a suffix of the page origin hostname

**Origin:**

The origin is the full scheme, hostname, and port. Examples:

- Development: `http://localhost:3000`
- Production: `https://passless.example.com`

Browser enforces strict matching: the page origin must exactly match `EXPECTED_ORIGIN` in the server configuration.

**User Handle:**

Each user is assigned a unique opaque byte string (user handle) that is not the username. This allows:

- Username changes without invalidating credentials
- Privacy: authenticators don't directly expose usernames
- Proper scope isolation between users

### 2.2 Registration Flow (Attestation Ceremony)

**Step 1: Client Requests Challenge**

```
POST /api/register/options
{ "username": "alice", "displayName": "Alice" }
```

**Server Actions:**

- Validate username format (email/alphanumeric)
- Generate random 32-byte challenge
- Store challenge in database with 5-minute TTL
- Create or retrieve user record with unique user handle

**Response:**

```json
{
  "challenge": "...",
  "rp": { "id": "localhost", "name": "WebAuthn Passwordless" },
  "user": { "id": "...", "name": "alice", "displayName": "Alice" },
  "pubKeyCredParams": [{ "alg": -7, "type": "public-key" }],
  "timeout": 60000,
  "attestation": "direct"
}
```

**Step 2: Browser Registration**

- User browser calls `navigator.credentials.create()` with challenge options
- Authenticator prompts user for consent/biometric
- Authenticator generates public/private key pair
- Authenticator creates attestation object containing:
  - Client data JSON (challenge, origin, type: "webauthn.create")
  - Attestation statement (proves authenticator genuineness)
  - Credential public key (COSE format)

**Step 3: Client Sends Response**

```
POST /api/register/verify
{ "id": "...", "rawId": "...", "response": { "clientDataJSON": "...", "attestationObject": "..." }, "type": "public-key" }
```

**Server Verification (Critical Security Checks):**

1. **Challenge Validation**
   - Decode `clientDataJSON` (base64 -> JSON)
   - Verify challenge matches stored challenge
   - Verify challenge has not expired

2. **Origin Validation**
   - Extract `origin` from decoded `clientDataJSON`
   - Verify origin matches `EXPECTED_ORIGIN` or allowed patterns
   - Rejects mismatched origins (e.g., attacker domain)

3. **RP ID Validation**
   - Extract RP ID hash from attestation object
   - Compute SHA-256 hash of configured `RP_ID`
   - Verify RP ID hash matches (prevents credential binding to wrong RP)

4. **Attestation Signature Verification**
   - Extract attestation statement from attestation object
   - Verify signature chain (attestation cert → root CA)
   - Ensures authenticator is genuine (optional in conveyance mode)

5. **Credential Storage**
   - Extract credential ID (randomly generated by authenticator)
   - Extract public key (COSE format)
   - Extract transports (USB, NFC, BLE, internal platform, etc.)
   - Store credential tied to user
   - Initialize signature counter to 0

**Response:**

- Session established (HttpOnly cookie)
- Redirect to protected `/app` route

### 2.3 Authentication Flow (Assertion Ceremony)

**Step 1: Client Requests Challenge**

```
POST /api/login/options
{ "username": "alice" }
```

**Server Actions:**

- Look up user by username
- Generate random 32-byte challenge
- Store challenge with user ID and 5-minute TTL
- Return list of user's registered credentials

**Response:**

```json
{
  "challenge": "...",
  "rpId": "localhost",
  "allowCredentials": [{ "id": "...", "type": "public-key", "transports": ["usb", "internal"] }],
  "timeout": 60000,
  "userVerification": "preferred"
}
```

**Step 2: Browser Authentication**

- User browser calls `navigator.credentials.get()` with challenge options
- Authenticator prompts user for biometric/PIN
- Authenticator retrieves private key (never exposed to browser)
- Authenticator computes assertion signature over client data
- Authenticator increments signature counter locally

**Step 3: Client Sends Response**

```
POST /api/login/verify
{ "id": "...", "rawId": "...", "response": { "clientDataJSON": "...", "authenticatorData": "...", "signature": "..." }, "type": "public-key" }
```

**Server Verification (Critical Security Checks):**

1. **Challenge Validation**
   - Decode `clientDataJSON`
   - Verify challenge matches stored challenge
   - Verify challenge has not expired

2. **Origin Validation**
   - Extract `origin` from `clientDataJSON`
   - Verify matches `EXPECTED_ORIGIN`

3. **RP ID Validation**
   - Extract RP ID hash from `authenticatorData`
   - Verify matches SHA-256 hash of configured RP ID

4. **User Presence Flag**
   - Check bits in `authenticatorData.flags`
   - User presence bit (0x01) must be set
   - User verification bit (0x04) checked if required

5. **Signature Counter Validation** (Replay Protection)
   - Extract counter from `authenticatorData`
   - Compare to stored counter for this credential
   - **MUST** be strictly greater than stored counter (prevents replay)
   - Update stored counter to new value
   - If counter not incremented: reject (possible cloned authenticator)

6. **Assertion Signature Verification**
   - Extract public key from stored credential
   - Verify signature over `clientDataJSON` + `authenticatorData`
   - Rejects forged assertions

**Response:**

- Session established
- Redirect to `/app` (authenticated dashboard)

### 2.4 Cryptographic Verification

#### Challenge-Response Protocol

- **Challenge:** 32-byte cryptographically random value
- **TTL:** 5 minutes (prevents stale challenges)
- **Prevents:** Replay attacks, CSRF, replay across registrations

#### Signature Scheme

- **Algorithm:** ECDSA with SHA-256 (ES256, COSE alg -7)
- **Curve:** P-256 (secp256r1)
- **Verification:** ECDSA signature must match public key
- **Prevents:** Forgery, replay, cross-credential attacks

#### RP ID Hash

- **Hash:** SHA-256(RP_ID)
- **Stored in:** Attestation object and authenticator data
- **Prevents:** Credential binding to wrong relying party

#### Signature Counter

- **Incremented:** On each successful assertion
- **Validated:** Must be strictly greater than previous value
- **Purpose:** Detects cloned authenticators (counter doesn't advance)
- **Side-Channel:** Counter wraps at 2^32; cloned authenticators detected even after wrap

## 3. Deployment & Accessibility

### 3.1 HTTPS Requirement

WebAuthn requires secure context:

- `http://localhost` is allowed (exception for development)
- All other origins must be HTTPS
- Self-signed certificates accepted in development
- Production uses Let's Encrypt or commercial CA

### 3.2 Cross-Device Access

**Named Tunnel (Recommended):**

- Fixed hostname persists across sessions
- Same RP ID across all devices
- Credentials registered on Device A work on Device B
- Suitable for production

**Quick Tunnels (Testing):**

- Hostname changes each run
- RP ID must be `localhost`
- Credentials tied to one hostname; re-register on new tunnel
- Suitable for quick testing

**Production:**

- Use fixed domain (e.g., `api.example.com`)
- Enable HTTPS with certificate
- RP ID = domain (no localhost)

### 3.3 Multi-Device Credential Portability

For same credential to work across devices:

1. **Same RP ID:** All devices must use same `RP_ID` value
2. **Same Origin:** All devices must access same scheme/hostname/port
3. **Same Username:** Authentication challenge specifies username → looks up user's credentials

If RP ID or origin changes, browser rejects credential (different binding).

## 6. Bonus: Cross-Device Credential Portability

### How RP ID Enables Multi-Device Use

The RP ID is the cryptographic binding point. During registration:

```
challenge + origin + RP_ID → used in client data hash → part of signed assertion
```

When authenticating:

```
RP_ID from request must match RP_ID used at registration
```

If RP ID changes (e.g., different hostname):

- Browser rejects credential (RP ID hash mismatch)
- User must re-register on new hostname

**Solution for multi-device:**

- Use **fixed, stable hostname** (e.g., `passless.example.com`)
- RP ID remains same across all devices
- Same credential works from any device

### Cross-Browser Credential Reuse

While credential itself is authenticator-specific:

- Public key in credential is format-agnostic (COSE)
- Same public key can be verified by any compliant server
- Credential ID (which authenticator to use) is unique per authenticator
- Browser/OS handles routing to correct authenticator

Example:

- Register on Device A with Platform Authenticator (Windows Hello) → Credential ID = X
- Register on Device B with Platform Authenticator (Face ID) → Credential ID = Y
- Authenticate on Device A → Browser uses Credential ID X → Windows Hello
- Authenticate on Device B → Browser uses Credential ID Y → Face ID

Multiple authenticators per user are supported; authentication challenge includes list of allowed credential IDs.

### Username-Based Account Linking

Users can link multiple devices to same account by:

1. Register on Device A as username `alice`
2. Register on Device B with same username `alice`
3. Both authenticators added to `alice`'s credential list
4. Authenticate as `alice` from any device; browser prompts which authenticator to use

This maintains single user account while supporting multiple credential authenticators.

## Appendix B: Protocol Verification Details

### Challenge Verification

**What:** Prevent replay attacks by ensuring same challenge can't be reused

**How:**

1. Generate random challenge
2. Store challenge in database
3. On verify, decode client data JSON
4. Check `challenge` field matches stored challenge
5. Delete challenge from database (single-use)
6. Check challenge age < 5 minutes (TTL)

**Attack Prevented:**

```
Attacker intercepts: curl ... -d '{"assertion": ...}'
Attacker replays: curl ... -d '{"assertion": ...}'
Server: Challenge already used → Rejection
```

### Origin Validation

**What:** Prevent phishing/cross-site attacks by binding credential to origin

**How:**

1. Configure `EXPECTED_ORIGIN` = `http://localhost:3000`
2. On verify, extract `origin` from decoded client data JSON
3. Verify `origin` matches exactly (scheme, host, port)
4. For tunnels, verify origin matches pattern (regex)

**Attack Prevented:**

```
Phisher sets up: phishing.com
Victim registers at: http://localhost:3000
Phisher tries to use credential at: https://phishing.com
Server: Origin mismatch → Rejection
```

### RP ID Hash Verification

**What:** Prevent credential binding to wrong relying party (RP collision)

**How:**

1. Configure `RP_ID` = `localhost`
2. Compute: `rpIdHash = SHA256(RP_ID)`
3. Extract `rpIdHash` from attestation/authenticator data
4. Verify extracted hash matches computed hash

**Attack Prevented:**

```
Alice uses: RP_ID = example.com
Bob runs malicious: RP_ID = example.com (same domain, different server)
Alice's credential cannot be used at Bob's server
(different server cert, credential tied to Alice's original server)
```

### Signature Counter Validation

**What:** Detect cloned/extracted authenticators using counter invariant

**How:**

1. Authenticator initializes: counter = 0
2. On each assertion, authenticator increments counter locally
3. Server stores counter = 0 initially
4. On first assertion: authenticator sends counter = 1
5. Server verifies: 1 > 0 (stored) → Accept, update stored to 1
6. On second assertion: authenticator sends counter = 2
7. Server verifies: 2 > 1 (stored) → Accept, update stored to 2
8. On replay of first assertion: authenticator sends counter = 1
9. Server verifies: 1 > 2 (stored) → REJECT

**Attack Prevented:**

```
Attacker clones authenticator's private key
Attacker's clone has counter = 0 (not incremented by original)
Original authenticator increments counter to 1, 2, 3, ...
Attacker's clone sends counter = 1 (same as original's first assertion)
Server: counter not incremented → Rejection
```

### Assertion Signature Verification

**What:** Ensure only holder of private key can authenticate

**How:**

1. Server stores public key from registration
2. On assertion, extract signature from response
3. Verify signature over: SHA256(clientDataJSON + authenticatorData)
4. Verify signature with stored public key (ECDSA P-256)

**Attack Prevented:**

```
Attacker intercepts: assertion response
Attacker modifies: changes username in clientDataJSON
Attacker sends modified assertion
Server: Signature verification fails (hash changed)
```

This design ensures passwordless auth achieves comparable or superior security to password-based auth through cryptographic binding and multi-factor verification.
